# Теоретическая подготовка к защите лабораторной работы №2
## Тема: Сетевая файловая система NFSv4

---

## Раздел 1. Основы NFS (Network File System)

### 1.1 Что такое NFS и его назначение

**NFS (Network File System)** — это протокол сетевой файловой системы, разработанный компанией Sun Microsystems, позволяющий клиентам монтировать удаленные файловые системы через сеть и работать с ними как с локальными .

**Ключевая особенность NFS:** Он предоставляет **прозрачный доступ** к файлам — клиентские приложения не знают, что файл находится на удаленном сервере. В отличие от FTP или HTTP, где файл сначала скачивается, NFS позволяет программам читать и записывать файлы по сети без их полного копирования.

### 1.2 История версий

| Версия | Год | Ключевые особенности |
|--------|-----|---------------------|
| NFSv2 | 1989 | Только UDP, ограничение размера файла 2 ГБ |
| NFSv3 | 1995 | Добавлена поддержка TCP, асинхронная запись, 64-битные файлы |
| NFSv4 | 2000 | Интеграция протоколов, единственный порт 2049, безопасность, ACL |
| NFSv4.1 | 2010 | Поддержка кластеризации (pNFS) |
| NFSv4.2 | 2016 | Поддержка Sparse files, клонирование, сжатие |

### 1.3 Сравнение NFSv3 и NFSv4

| Характеристика | NFSv3 | NFSv4 |
|----------------|-------|-------|
| Протоколы | TCP и UDP | Только TCP (рекомендован) |
| Порт для firewall | Множество динамических портов | Один порт - 2049  |
| Зависимость от rpcbind | Да (порт 111) | Нет |
| Безопасность | AUTH_SYS (слабая) | Kerberos (RPCSEC_GSS) |
| Состояние соединения | Stateless (без состояния) | Stateful (с сохранением состояния) |
| Поддержка ACL | Нет | Да |
| Операции | Множество отдельных вызовов | Составные операции (COMPOUND) |

**Ключевое преимущество NFSv4 для системных администраторов:** Значительное упрощение настройки межсетевых экранов (firewall). Если NFSv3 требует открытия нескольких динамических портов и службы rpcbind, то для NFSv4 достаточно открыть только TCP-порт 2049 .

## Раздел 2. Архитектура и компоненты NFS

### 2.1 Основные компоненты

1. **NFS-сервер** — экспортирует (предоставляет) свои каталоги для доступа по сети
2. **NFS-клиент** — монтирует удаленные каталоги и работает с ними
3. **RPC (Remote Procedure Call)** — механизм удаленного вызова процедур, лежащий в основе NFS 
4. **rpcbind (portmapper)** — служба, сопоставляющая RPC-программы с портами (не нужна для NFSv4, но может требоваться для совместимости) 

### 2.2 RPC и динамические порты в NFSv3

В NFSv3 и более старых версиях используется несколько служб с динамически назначаемыми портами:
- **portmapper (rpcbind)** — порт 111 (фиксированный)
- **rpc.mountd** — порт динамический (отвечает за первоначальное монтирование)
- **rpc.statd** — порт динамический (восстановление после сбоев)
- **lockd (nlockmgr)** — порт динамический (блокировка файлов)

Это создает проблемы при настройке firewall, так как порты неизвестны заранее . В современных системах можно зафиксировать эти порты .

### 2.3 Как NFSv4 упрощает работу с firewall

**В NFSv4 все операции выполняются через единый порт 2049/TCP** . Это означает, что для работы NFSv4 через firewall достаточно добавить всего одно правило:

```bash
# Для сервера (разрешить входящие соединения от клиента)
iptables -A INPUT -s <IP_клиента> -p tcp --dport 2049 -j ACCEPT

# Для клиента (разрешить исходящие соединения к серверу)
iptables -A OUTPUT -d <IP_сервера> -p tcp --dport 2049 -j ACCEPT
```

Именно поэтому в задании NFSv4 позиционируется как "firewall-friendly" протокол .

## Раздел 3. Настройка NFS-сервера

### 3.1 Bind-монтирование (mount --bind)

**Bind-монтирование** — это механизм, позволяющий сделать существующий каталог доступным в другой точке монтирования без физического копирования файлов.

```bash
mount --bind /исходный_каталог /целевой_каталог
```

**Применение в работе:** Мы использовали bind-монтирование для подключения системных каталогов `/etc/snmp` и `/var/log` в `/srv/nfs/` — каталог, который затем экспортируется через NFS.

**Важно:** Для сохранения после перезагрузки bind-монтирования необходимо добавить в `/etc/fstab`:
```bash
/источник    /цель    none    bind    0 0
```

### 3.2 Файл /etc/exports — управление экспортами

Файл `/etc/exports` определяет, какие каталоги экспортируются и для каких клиентов .

**Синтаксис:**
```bash
<каталог>    <клиент>(<опции>)
```

**Основные опции:**

| Опция | Значение |
|-------|----------|
| `ro` | Только чтение |
| `rw` | Чтение и запись |
| `sync` | Запись на диск до подтверждения клиенту |
| `async` | Подтверждение до фактической записи (быстрее, но рискованнее) |
| `no_subtree_check` | Отключает проверку поддеревьев (повышает производительность) |
| `fsid=0` | Указывает корневой каталог для NFSv4 (обязательно)  |
| `insecure` | Разрешает подключения с непривилегированных портов (>1024) |
| `no_root_squash` | Отключает преобразование root-клиента в анонимного пользователя |
| `root_squash` | Преобразует root-клиента в анонимного пользователя (по умолчанию)  |

### 3.3 Важность опции fsid=0

Для NFSv4 **обязательно** указание опции `fsid=0` для корневого каталога экспорта . Это определяет "псевдо-корень" файловой системы NFSv4. Клиент обращается к экспортам относительно этого корня:

- На сервере: `/srv/nfs (fsid=0)`
- На клиенте: монтирование `:/snmp` (а не `:/srv/nfs/snmp`)

### 3.4 Команды управления экспортами

```bash
# Просмотр текущих экспортов
exportfs -v

# Перечитать /etc/exports и применить изменения
exportfs -arv    # -a: все экспорты, -r: перечитать, -v: подробно

# Временный экспорт каталога
exportfs -o rw <клиент>:<каталог>

# Отмена экспорта
exportfs -u <клиент>:<каталог>
```

## Раздел 4. Настройка брандмауэра (iptables)

### 4.1 Особенности NFSv4 для firewall

**Для NFSv4 требуется только порт 2049/TCP** . Это коренное отличие от NFSv3, требующего нескольких портов.

**Правила для сервера NFSv4:**
```bash
# Разрешить входящие соединения NFS от клиента
iptables -A INPUT -s <IP_клиента> -p tcp --dport 2049 -j ACCEPT
```

### 4.2 Важность порядка правил

**Критическое замечание:** В iptables правила обрабатываются последовательно, сверху вниз. Если в конце цепочки INPUT есть правило `REJECT all` (как в CentOS 6 по умолчанию), новые правила, добавленные с `-A` (append), попадают после него и никогда не срабатывают .

**Решение:** Использовать `-I` (insert) для вставки правил в начало цепочки или на конкретную позицию:
```bash
# Вставить перед правилом REJECT (на позицию 6)
iptables -I INPUT 6 -s <IP> -p tcp --dport 2049 -j ACCEPT
```

### 4.3 Для справки: правила для NFSv3

Если бы использовалась NFSv3, потребовалось бы значительно больше правил :

```bash
# Для NFSv3 требуется открыть:
iptables -A INPUT -p tcp --dport 111 -j ACCEPT        # rpcbind
iptables -A INPUT -p tcp --dport 2049 -j ACCEPT       # NFS
iptables -A INPUT -p tcp --dport 4045 -j ACCEPT       # lockd
iptables -A INPUT -p tcp --dport 4046 -j ACCEPT       # statd
iptables -A INPUT -p tcp --dport 4047 -j ACCEPT       # mountd
```

Это наглядно демонстрирует преимущество NFSv4 перед старыми версиями.

### 4.4 Сохранение правил iptables

В CentOS 6/RHEL 6:
```bash
service iptables save    # Сохраняет правила в /etc/sysconfig/iptables
service iptables restart # Применяет сохраненные правила
```

## Раздел 5. Настройка NFS-клиента

### 5.1 Монтирование NFS-шары

**Базовая команда монтирования:**
```bash
mount -t nfs4 <сервер>:/путь /точка_монтирования
```

**Важное замечание о версиях:** В NFSv4 путь указывается относительно корня, заданного опцией `fsid=0` на сервере .

### 5.2 Проблема совместимости версий

**Проблема:** Современные дистрибутивы Linux (Ubuntu 22.04) по умолчанию пытаются использовать NFSv4.2, в то время как старые серверы (CentOS 6.10) поддерживают только NFSv4.0.

**Симптом:** Команда `mount` зависает, так как клиент пытается согласовать версию протокола, а сервер не отвечает на запросы новых версий.

**Решение:** Явное указание версии протокола:
```bash
mount -t nfs4 -o vers=4.0 <сервер>:/путь /точка_монтирования
```

### 5.3 Опции монтирования

| Опция | Значение |
|-------|----------|
| `vers=4.0` | Явно указать версию NFSv4.0 |
| `_netdev` | Сетевое устройство — монтировать только после поднятия сети |
| `ro` | Монтировать только для чтения |
| `rw` | Монтировать для чтения и записи |
| `hard` | При сбое сервера операция зависает до восстановления (по умолчанию) |
| `soft` | При сбое операция завершается с ошибкой |
| `intr` | Разрешить прерывание зависших операций |
| `timeo=n` | Таймаут в десятых долях секунды |
| `retrans=n` | Количество повторных попыток |

### 5.4 Автоматическое монтирование — /etc/fstab

Для автоматического монтирования при загрузке системы записи добавляются в `/etc/fstab`:

```bash
<сервер>:/путь    /точка_монтирования    nfs    опции    0 0
```

**Пример из работы:**
```bash
172.16.8.149:/snmp   /mnt/centos6-server/snmp   nfs   vers=4.0,_netdev   0 0
172.16.8.149:/log    /mnt/centos6-server/log    nfs   vers=4.0,_netdev,ro   0 0
```

**Ключевые опции для fstab:**
- `_netdev` — критически важна для сетевых файловых систем 
- `vers=4.0` — фиксирует версию протокола

**Проверка fstab:**
```bash
mount -a    # Монтирует все файловые системы из fstab
```

## Раздел 6. Права доступа и безопасность

### 6.1 Два уровня проверки прав доступа

При работе с NFS существует **два независимых уровня** проверки прав доступа:

1. **Уровень NFS (экспорты)** — задается в `/etc/exports` опциями `ro`/`rw`
2. **Уровень файловой системы** — реальные права (UNIX permissions) на сервере

**Важно:** Даже если в exports указан `rw`, запись может быть невозможна из-за прав файловой системы на сервере.

### 6.2 Причина ошибки Permission denied

В ходе работы возникла ошибка `Permission denied` при попытке записи в каталог `snmp`, несмотря на режим `rw` в exports.

**Причина:** Каталог `/srv/nfs/snmp` (bind-монтирование `/etc/snmp`) имел права `drwxr-xr-x` (755), что разрешает запись только владельцу (root). Пользователь `tech` на клиенте имел другой UID и не мог писать.

**Решение:** Изменение прав на сервере:
```bash
chmod 777 /srv/nfs/snmp/    # Открыть полный доступ (для лабораторной работы)
```

### 6.3 Root squashing

**Root squashing** — механизм безопасности, при котором пользователь root на клиенте отображается в анонимного пользователя (обычно `nobody` или `nfsnobody`) на сервере . Это предотвращает получение привилегий root на сервере через NFS.

- `root_squash` — включено по умолчанию
- `no_root_squash` — отключает это преобразование (не рекомендуется для production)

### 6.4 Отображение пользователей (UID/GID)

NFS использует числовые идентификаторы UID и GID для проверки прав доступа. Для корректной работы необходимо, чтобы на сервере и клиенте существовали пользователи с одинаковыми UID.

## Раздел 7. Диагностика и отладка

### 7.1 Проверка работы сервера

```bash
# Проверка, слушает ли сервер порт 2049
netstat -tulpn | grep 2049

# Просмотр RPC-сервисов
rpcinfo -p localhost

# Проверка экспортов
exportfs -v
showmount -e localhost
```

### 7.2 Проверка доступности порта с клиента

```bash
# Быстрая проверка TCP-порта
nc -zv <IP_сервера> 2049

# Более детальная проверка
telnet <IP_сервера> 2049
```

### 7.3 Логирование

| ОС | Файлы логов |
|----|-------------|
| CentOS/RHEL 6 | `/var/log/messages` |
| Ubuntu/Debian | `/var/log/syslog` |

```bash
# Просмотр логов NFS в реальном времени
tail -f /var/log/messages | grep -i nfs   # CentOS
tail -f /var/log/syslog | grep -i nfs     # Ubuntu
```

### 7.4 Типичные ошибки и их причины

| Ошибка | Вероятная причина |
|--------|-------------------|
| `mount.nfs: No route to host` | Firewall блокирует порт 2049  |
| `Permission denied` | Проблема с правами на сервере или экспортами  |
| Зависание команды mount | Несовместимость версий NFS |
| `mount.nfs: access denied by server` | Клиент не разрешен в /etc/exports  |
| `mount.nfs: Connection timed out` | Сетевые проблемы или firewall блокирует порты |

## Раздел 8. Практические аспекты выполнения работы

### 8.1 Схема взаимодействия компонентов

```
[Сервер CentOS 6.10: 172.16.8.149]
    │
    ├── /etc/snmp ──bind──> /srv/nfs/snmp ──export──> NFSv4
    ├── /var/log ──bind──> /srv/nfs/log ──export──> NFSv4
    │
    ├── /etc/exports: fsid=0, rw/ro для 172.16.8.169
    ├── iptables: порт 2049 открыт для 172.16.8.169
    │
    └─────────────── NFSv4 (порт 2049) ───────────────┐
                                                        │
[Клиент Ubuntu 22.04: 172.16.8.169]                   │
    │                                                   │
    ├── /mnt/centos6-server/snmp ◄─────────────────────┘
    ├── /mnt/centos6-server/log
    │
    ├── fstab: vers=4.0,_netdev
    └── Проверка: запись в snmp, только чтение в log
```

### 8.2 Основные сложности и их решение

| Сложность | Решение |
|-----------|---------|
| Зависание mount | Указать `vers=4.0` |
| No route to host | Открыть порт 2049 в iptables |
| Правило REJECT блокирует | Вставить ACCEPT перед REJECT (опция -I) |
| Permission denied | Изменить права на сервере (`chmod 777`) |

### 8.3 Проверка работоспособности

```bash
# Проверка монтирования
df -h | grep nfs
mount | grep nfs

# Проверка записи (каталог snmp)
touch /mnt/.../snmp/test.txt
echo "test" > /mnt/.../snmp/test.txt

# Проверка чтения (каталог log — только чтение)
ls -la /mnt/.../log/
# Попытка записи должна вызвать ошибку
touch /mnt/.../log/test.txt    # Read-only file system
```

## Заключение

NFSv4 представляет собой значительный шаг вперед по сравнению с предыдущими версиями протокола. Основные выводы по выполненной работе:

1. **Упрощение firewall** — использование единственного порта 2049 делает NFSv4 значительно более удобным для настройки в защищенных сетях .

2. **Важность совместимости версий** — при работе с разными операционными системами необходимо явно указывать версию протокола.

3. **Два уровня прав доступа** — недостаточно настроить экспорты, необходимо также обеспечить соответствующие права на файловой системе сервера.

4. **Правильный порядок правил firewall** — критически важен для корректной работы, особенно при наличии общих запрещающих правил.

5. **Bind-монтирование** — мощный инструмент для экспортирования существующих системных каталогов без их перемещения.
