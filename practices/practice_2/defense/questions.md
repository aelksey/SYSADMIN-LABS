## Вопросы для защиты лабораторной работы №2 (NFSv4)

---

### Блок 1: Теоретические вопросы по NFS

#### 1. Что такое NFS и для чего он используется?

**Ответ:** NFS (Network File System) — это сетевой протокол, позволяющий монтировать удаленные файловые системы через сеть так, как если бы они были локальными . Основное отличие от FTP или HTTP в том, что NFS предоставляет прямой доступ к файловой системе, а не просто передачу файлов. Это позволяет программам работать с удаленными файлами без их предварительного скачивания .

---

#### 2. В чем основные отличия NFSv3 от NFSv4?

**Ответ:** NFSv4 имеет ряд важных преимуществ по сравнению с NFSv3 :

| Характеристика | NFSv3 | NFSv4 |
|----------------|-------|-------|
| Поддержка UTF-8 | Нет | Да |
| ACL (списки доступа) | Нет | Да |
| Блокировка диапазона байтов | Нет | Да |
| Состояние соединения | Без состояния (stateless) | С сохранением состояния |
| RPC-зависимость | Требует rpcbind | Не требует rpcbind |
| Работа с файрволом | Сложно (много динамических портов) | Просто (один порт 2049) |

Кроме того, NFSv4 использует единый порт 2049 для всех операций, что упрощает настройку firewall , в то время как NFSv3 использует несколько динамических портов, которые необходимо фиксировать.

---

#### 3. Почему NFSv4 считается более firewall-friendly?

**Ответ:** NFSv3 и более старые версии используют несколько динамических портов для различных сервисов (portmapper, mountd, lockd, statd), что затрудняет настройку firewall . NFSv4, напротив, использует только один порт — 2049/TCP — для всех операций, что позволяет легко настроить разрешающие правила. Именно поэтому в задании требовалось ограничить доступ к портам rpcbind (111) — это legacy-требование, не обязательное для чистой NFSv4.

### Блок 2: Настройка сервера (CentOS 6.10)

#### 4. Для чего используется команда `mount --bind`? Как она работает?

**Ответ:** `mount --bind` позволяет привязать существующий каталог к другой точке монтирования, делая его содержимое доступным в двух местах одновременно. В нашей работе мы использовали:

```bash
sudo mount --bind /etc/snmp /srv/nfs/snmp
sudo mount --bind /var/log /srv/nfs/log
```

Это позволило "экспортировать" системные каталоги `/etc/snmp` и `/var/log` через NFS, не перемещая их физически. При этом изменения в одном каталоге автоматически отображаются в другом. Для сохранения после перезагрузки такие привязки добавляются в `/etc/fstab`.

---

#### 5. Какие опции в `/etc/exports` вы использовали и что они означают?

**Ответ:** В файле `/etc/exports` были добавлены следующие строки:

```bash
/srv/nfs        172.16.8.169(rw,sync,fsid=0,no_subtree_check,insecure)
/srv/nfs/snmp   172.16.8.169(rw,sync,no_subtree_check,insecure)
/srv/nfs/log    172.16.8.169(ro,sync,no_subtree_check,insecure)
```

Значение опций:

- **rw/ro** — режим чтения-записи или только чтения
- **sync** — запись на диск до подтверждения клиенту (безопаснее, но медленнее)
- **fsid=0** — определяет корневой каталог для NFSv4 (обязательная опция)
- **no_subtree_check** — отключает проверку поддеревьев (повышает производительность)
- **insecure** — разрешает подключения с непривилегированных портов (>1024) ; необходимо для современных клиентов
- **no_root_squash** — (не использовали, но упомянуть) отключает преобразование root-пользователя клиента в анонимного пользователя на сервере

#### 6. Как проверить, какие каталоги экспортируются через NFS?

**Ответ:** Существует несколько команд для проверки:

```bash
# Просмотр текущих экспортов
sudo exportfs -v

# Просмотр с клиента
showmount -e <IP_сервера>

# Просмотр с сервера через rpcinfo
rpcinfo -p localhost
```

#### 7. В чем сложность настройки iptables для NFSv4 на CentOS 6?

**Ответ:** Основная сложность заключается в **порядке правил** . В стандартной конфигурации CentOS 6 в конце цепочки INPUT находится правило `REJECT all`, которое запрещает все, что не было явно разрешено выше. При добавлении новых правил с помощью `-A` (append) они попадают в конец списка, после запрещающего правила, и никогда не срабатывают.

**Решение:** Использовать `-I` (insert) с указанием позиции, чтобы вставить правило перед запрещающим:

```bash
sudo iptables -I INPUT 6 -s 172.16.8.169 -p tcp --dport 2049 -j ACCEPT
```

### Блок 3: Настройка клиента (Ubuntu 22.04)

#### 8. Почему команда `mount` зависала и как это исправили?

**Ответ:** Команда `mount` зависала из-за **несовместимости версий NFS**. Клиент Ubuntu 22.04 по умолчанию пытается использовать NFSv4.2, который не поддерживается сервером CentOS 6.10 (поддерживает только NFSv4.0). Процесс автоматического согласования версий зависал в ожидании ответа от сервера.

**Решение:** Явное указание версии протокола в опциях монтирования:

```bash
sudo mount -t nfs4 -o vers=4.0 172.16.8.149:/snmp /mnt/.../snmp
```

---

#### 9. Что означает опция `_netdev` в `/etc/fstab`?

**Ответ:** Опция `_netdev` указывает системе, что монтируемое устройство является сетевым. Это гарантирует, что система будет ждать поднятия сети перед попыткой монтирования NFS-шары при загрузке. Без этой опции монтирование может провалиться, так как сеть еще не инициализирована . Пример использования в `/etc/fstab`:

```bash
172.16.8.149:/snmp   /mnt/centos6-server/snmp   nfs   vers=4.0,_netdev   0 0
```

---

#### 10. Как проверить, что NFS-шара смонтирована?

**Ответ:** Существует несколько способов проверки:

```bash
# Просмотр всех смонтированных файловых систем
df -h | grep nfs

# Более детальная информация
mount | grep nfs

# Просмотр конкретного каталога
ls -la /mnt/centos6-server/snmp/

# Проверка записи (если разрешено)
touch /mnt/centos6-server/snmp/test_file.txt
```

### Блок 4: Проблемы и решения

#### 11. В чем причина ошибки "No route to host" при монтировании?

**Ответ:** Ошибка "No route to host" означает, что клиент не может установить TCP-соединение с портом 2049 на сервере. При этом сам сервер доступен по ping. Наиболее вероятные причины :

1. **Firewall на сервере** блокирует порт 2049
2. Неправильный порядок правил в iptables (правило REJECT находится выше ACCEPT)
3. Служба NFS не запущена на сервере

В нашем случае проблема была в отсутствии разрешающего правила для порта 2049.

---

#### 12. Почему возникла ошибка "Permission denied" при попытке записи, хотя в exports указан режим rw?

**Ответ:** Эта ошибка возникает из-за конфликта между настройками экспорта NFS и фактическими правами доступа на файловой системе сервера .

Несмотря на режим `rw` в `/etc/exports`, реальная файловая система (в нашем случае `/etc/snmp`) имела права `drwxr-xr-x` (755), что разрешает запись только владельцу (root). Пользователь `tech` на клиенте имел другой UID и не мог писать.

**Решение:** Изменение прав на сервере:
```bash
sudo chmod 777 /srv/nfs/snmp/
```

**Альтернатива:** Настройка отображения пользователей через `anonuid`/`anongid` или создание пользователя с одинаковым UID на обоих серверах.

---

#### 13. Что такое root squashing и как оно влияет на доступ?

**Ответ:** Root squashing — это механизм безопасности NFS, который по умолчанию отображает пользователя root на клиенте в анонимного пользователя (обычно `nobody`) на сервере . Это предотвращает получение привилегий root на сервере через NFS.

В нашей работе это не было проблемой, так как пользователь `tech` не является root. Если бы требовался доступ от имени root, пришлось бы использовать опцию `no_root_squash` (что не рекомендуется для production-сред).

---

#### 14. Какие проблемы могут возникнуть из-за SELinux при работе с NFS?

**Ответ:** SELinux в режиме `Enforcing` может блокировать запись через NFS даже при правильно настроенных правах файловой системы и экспортов . Для разрешения необходимы команды:

```bash
# Включение записи через NFS
sudo setsebool -P nfs_export_all_rw on
sudo setsebool -P nfs_export_all_ro on
```

Для диагностики можно временно отключить SELinux (`setenforce 0`), но в production-среде следует настраивать правильные политики.

### Блок 5: Практические аспекты

#### 15. Как настроить автоматическое монтирование NFS-шары после перезагрузки?

**Ответ:** Для автоматического монтирования необходимо добавить записи в `/etc/fstab` на клиенте :

```bash
172.16.8.149:/snmp   /mnt/centos6-server/snmp   nfs   vers=4.0,_netdev   0 0
172.16.8.149:/log    /mnt/centos6-server/log    nfs   vers=4.0,_netdev,ro   0 0
```

Ключевые опции:
- `vers=4.0` — фиксирует версию протокола
- `_netdev` — указывает на сетевое устройство
- `ro` — монтирование только для чтения (для каталога log)

После добавления записей проверьте командой `sudo mount -a`.

---

#### 16. Что будет, если перезагрузить сервер NFS во время работы клиента?

**Ответ:** При перезагрузке сервера NFS клиентское приложение, обращающееся к смонтированной шаре, временно "зависнет" в ожидании ответа от сервера. После восстановления сервера соединение должно автоматически восстановиться, если использовались опции `hard` (по умолчанию) и `intr` (разрешает прерывание). В нашей работе это требование пункта 7 — "доступность должна восстанавливаться автоматически".

---

#### 17. В чем отличие монтирования с указанием `nfs4` от обычного `nfs`?

**Ответ:** Команда `mount -t nfs4` явно указывает использование NFSv4 протокола. При использовании `mount -t nfs` система может автоматически определить версию, но в некоторых случаях (как в нашей работе) это приводит к проблемам из-за попытки использования более новых версий . Явное указание версии через опцию `vers=4.0` является наиболее надежным подходом при работе со старыми серверами.

---

#### 18. Какие логи нужно проверять при диагностике проблем NFS?

**Ответ:** На разных системах логи находятся в разных местах:

- **CentOS/RHEL 6:** `/var/log/messages`
- **Ubuntu:** `/var/log/syslog`

Просмотр логов NFS:
```bash
# На CentOS
sudo tail -f /var/log/messages | grep -i nfs

# На Ubuntu
sudo tail -f /var/log/syslog | grep -i nfs

# Проверка статуса RPC
rpcinfo -p localhost
```

### Блок 6: Синтез и обобщение

#### 19. Опишите полную схему взаимодействия компонентов в выполненной работе.

**Ответ:** Полная схема включает следующие компоненты и этапы:

1. **Сервер CentOS 6.10 (172.16.8.149):**
   - Bind-монтирование `/etc/snmp` → `/srv/nfs/snmp` и `/var/log` → `/srv/nfs/log`
   - Настройка `/etc/exports` с опциями для NFSv4
   - Запуск сервисов `nfs` и `rpcbind`
   - Настройка iptables (порты 111, 2049)

2. **Клиент Ubuntu 22.04 (172.16.8.169):**
   - Установка `nfs-common`
   - Создание точек монтирования `/mnt/.../snmp` и `/mnt/.../log`
   - Монтирование с опцией `vers=4.0`
   - Настройка `/etc/fstab` для автоматического монтирования

3. **Firewall:** Правила iptables на сервере разрешают доступ к портам 111 и 2049 только с IP клиента

4. **Проверка:** Создание файлов в `snmp` (rw), попытка записи в `log` (ro) — ошибка

#### 20. Какие выводы можно сделать по итогам работы?

**Ответ:** Основные выводы по выполненной работе:

1. **Совместимость версий критична** — при работе с разными ОС необходимо явно указывать версию NFS протокола.
2. **NFSv4 упрощает работу с firewall** благодаря использованию одного порта 2049.
3. **Bind-монтирование** — удобный способ экспортировать существующие каталоги без их перемещения.
4. **Права доступа** на экспортируемые каталоги должны быть настроены отдельно от опций экспорта.
5. **Порядок правил в iptables** имеет значение — разрешающие правила должны быть выше запрещающих.
6. **Автоматическое монтирование** требует правильной настройки `/etc/fstab` с опцией `_netdev`.
