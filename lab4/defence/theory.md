## 1. Процессы в Linux

### 1.1. Определение процесса
- **Процесс** – это экземпляр выполняемой программы, имеющий:
  - Уникальный идентификатор (PID – Process ID)
  - Собственное адресное пространство (код, данные, стек)
  - Набор открытых файлов и файловых дескрипторов
  - Окружение (переменные среды)
  - Текущую рабочую директорию
  - Счётчик команд и регистры процессора

### 1.2. PID и PPID
- **PID** – целое положительное число, уникальное для каждого процесса в системе.
- **PPID** – PID родительского процесса (parent process). Первым процессом в системе является `init` (PID=1) или `systemd`.

### 1.3. Состояния процесса
- **R (Running)** – выполняется или готов к выполнению.
- **S (Sleeping)** – ожидает события (ввод-вывод, сигнал).
- **D (Uninterruptible sleep)** – ожидание ввода-вывода, не реагирует на сигналы.
- **Z (Zombie)** – процесс завершился, но родитель не прочитал его код возврата.
- **T (Stopped)** – приостановлен (обычно по сигналу SIGSTOP или SIGTSTP).

### 1.4. Просмотр процессов
- `ps aux` – подробный список всех процессов.
- `ps -ef` – стандартный формат.
- `pgrep -f worker` – поиск PID по имени (с поддержкой регулярных выражений).
- `top`, `htop` – динамический просмотр.

### 1.5. Файловая система `/proc`
- Виртуальная файловая система, предоставляющая информацию о процессах и системе.
- `/proc/[PID]/` – каталог процесса с PID.
  - `cmdline` – командная строка запуска процесса (аргументы разделены нулевыми символами).
  - `status` – статус, память, PPID, UID и т.д.
  - `stat` – статистика в машинно-читаемом формате.
  - `fd/` – открытые файловые дескрипторы.
- **Пример чтения** `cmdline`:
  ```bash
  tr '\0' ' ' < /proc/12345/cmdline
  ```
  Замена нулей пробелами для удобочитаемости.

## 2. Сигналы

### 2.1. Определение
Сигналы – асинхронные уведомления, посылаемые процессу для уведомления о событиях (ошибка, запрос завершения, пользовательское действие).

### 2.2. Основные сигналы
| Сигнал | Номер | Действие по умолчанию | Описание |
|--------|-------|----------------------|-----------|
| SIGHUP | 1     | Завершение           | Закрытие терминала (отбой) |
| SIGINT | 2     | Завершение           | Прерывание с клавиатуры (Ctrl+C) |
| SIGQUIT| 3     | Завершение + дамп    | Выход с клавиатуры (Ctrl+\) |
| SIGKILL| 9     | Завершение (нельзя перехватить) | Безусловное убийство |
| SIGTERM| 15    | Завершение           | Запрос на завершение (можно обработать) |
| SIGSTOP| 19    | Остановка (нельзя перехватить) | Приостановка процесса |
| SIGUSR1| 10    | Завершение           | Пользовательский сигнал 1 |
| SIGUSR2| 12    | Завершение           | Пользовательский сигнал 2 |

### 2.3. Перехват сигналов в Bash (`trap`)
- **Синтаксис**: `trap 'команды' сигналы`
- Пример:
  ```bash
  trap 'echo "Получен SIGINT, но игнорирую" >> log' SIGINT
  ```
- В лабораторной работе `yakimenko.sh` перехватывает `SIGINT` и `SIGQUIT`, пишет сообщение в `observer.log` и продолжает работу (сигнал игнорируется).

### 2.4. Отправка сигналов
- `kill -SIGTERM 1234` (или `kill -15 1234`)
- `kill -9 1234` (безусловное убийство)
- `killall worker_1.sh` – по имени процесса.

## 3. Именованные каналы (FIFO)

### 3.1. Определение
**FIFO** (First In – First Out) – специальный файл, используемый для однонаправленной передачи данных между процессами. Работает как очередь: записанные данные читаются в том же порядке.

### 3.2. Создание
```bash
mkfifo /tmp/myfifo
```

### 3.3. Отличие от обычного pipe (`|`)
- Обычный pipe – анонимный, существует только между родственными процессами (например, в конвейере).
- Именованный канал – файл в ФС, может использоваться любыми процессами, даже не связанными родством.

### 3.4. Использование в лабораторной работе
- **Воркер** создаёт два канала:
  - `/tmp/workers/run/PID.pid` – для приёма команд от диспетчера.
  - `/tmp/workers/run/PID_reply.pid` – для отправки ответов диспетчеру.
- **Диспетчер** пишет в `PID.pid` команды (`how much?`, `work twice!`), а читает ответ из `PID_reply.pid`.

### 3.5. Опасности и особенности
- Чтение из пустого FIFO блокирует процесс до появления данных.
- Запись в FIFO без читающего процесса – блокировка (или ошибка, если не использовать O_NONBLOCK).
- Для избежания блокировок используют **отдельные каналы** на каждое направление и **таймауты при чтении** (`read -t`).

## 4. Запуск процессов в фоне и `nohup`

### 4.1. Фоновый режим
- Добавление `&` в конце команды запускает процесс в фоне:
  ```bash
  ./worker.sh &
  ```
- Процесс продолжает работать, но при закрытии терминала получит `SIGHUP` и завершится.

### 4.2. `nohup` – защита от SIGHUP
- `nohup` игнорирует сигнал `SIGHUP`, позволяя процессу продолжить работу после выхода из терминала.
- Использование:
  ```bash
  nohup ./worker.sh > /dev/null 2>&1 &
  ```
- Перенаправление stdout/stderr в `/dev/null` необходимо, чтобы не создавать файл `nohup.out`.

### 4.3. Почему `nohup` важен в лабораторной работе?
- `yakimenko.sh` запускает воркеров через `cron`. Cron выполняется в окружении без терминала, и без `nohup` воркеры могли бы завершиться сразу.
- Даже в интерактивной сессии `nohup` гарантирует, что воркеры не умрут при закрытии терминала.

## 5. Планировщик cron

### 5.1. Что такое cron?
- Демон `crond` запускает задачи по расписанию, описанным в **crontab**.
- Каждый пользователь может иметь свой crontab.

### 5.2. Формат строки crontab
```
* * * * * команда
│ │ │ │ │
│ │ │ │ └── день недели (0-7, 0=воскресенье)
│ │ │ └──── месяц (1-12)
│ │ └────── день месяца (1-31)
│ └──────── час (0-23)
└────────── минута (0-59)
```
- `*` означает «все возможные значения».
- `*/5` – каждые 5 единиц.

### 5.3. Настройка в лабораторной работе
- Задача: запускать `yakimenko.sh` каждую минуту.
- Запись в crontab:
  ```bash
  * * * * * /home/tech/lab4/yakimenko.sh
  ```
- Команда `crontab -e` открывает редактор для правки.

### 5.4. Обеспечение интервала не менее 59 секунд
- В `yakimenko.sh` замеряется время выполнения, и если оно меньше 59 секунд, скрипт досыпает оставшееся время.
- Это гарантирует, что следующий запуск cron'ом не произойдёт раньше, чем через минуту (хотя cron сам выдерживает интервал, но требование есть в задании).

## 6. Взаимодействие процессов через FIFO (логика лабораторной)

### 6.1. Диспетчер (`yakimenko.sh`)
1. Читает список путей к воркерам из конфигурационного файла `/tmp/workers/kafedra.conf`.
2. Для каждого воркера:
   - Ищет его PID, перебирая `/proc/[0-9]*/cmdline` (по имени скрипта).
   - Если не найден – запускает через `nohup`.
   - Если найден:
     - Отправляет `how much?` в `PID.pid`.
     - Читает ответ (исходное время и оставшееся) из `PID_reply.pid`.
     - Вычисляет порог 70% от исходного времени.
     - Если оставшееся время < 70% → с вероятностью 0.3 отправляет `work twice!`.
3. Обрабатывает сигналы `SIGINT` и `SIGQUIT` через `trap`.

### 6.2. Воркер (`template_worker.sh`)
1. Создаёт два канала: `PID.pid` и `PID_reply.pid`.
2. Запоминает случайное время работы (60–600 секунд).
3. В основном цикле (пока время не истекло):
   - Пытается прочитать команду из `PID.pid` с таймаутом 1 секунда (`read -t 1`).
   - Если команда `how much?` – отвечает двумя числами в `PID_reply.pid`.
   - Если `work twice!` – удваивает оставшееся время и пишет в лог.
   - Уменьшает счётчик оставшегося времени на 1, спит 1 секунду.
4. После завершения цикла удаляет каналы и пишет итоговый отчёт.

### 6.3. Почему воркер не использует фоновый процесс?
- В первом варианте использовался фоновый процесс для чтения канала, но это приводило к проблеме синхронизации переменной `REMAINING` (копировалась при запуске фонового процесса).
- Исправленный вариант – один процесс, чтение с таймаутом – позволяет обновлять `REMAINING` напрямую.

## 7. Важные детали реализации в Bash

### 7.1. Работа с файловыми дескрипторами
- `exec 3< "$IN_FIFO"` – открывает FIFO на чтение и связывает с дескриптором 3.
- `exec 4> "$OUT_FIFO"` – открывает на запись, дескриптор 4.
- Затем `read -r cmd <&3` и `echo ... >&4` обращаются к дескрипторам.
- Закрытие: `exec 3<&-`, `exec 4>&-`.

### 7.2. Функции в Bash
- `log() { ... }` – инкапсулирует запись в лог с временной меткой.
- `find_pid_by_script()` – возвращает PID по имени скрипта.

### 7.3. Генерация случайных чисел
- `$RANDOM` – псевдослучайное число от 0 до 32767.
- `TOTAL_TIME=$(( RANDOM % 541 + 60 ))` – диапазон 60..600.
- `if [ $(( RANDOM % 10 )) -lt 3 ]` – вероятность 0.3.

### 7.4. Обработка строк в cmdline
- В `/proc/PID/cmdline` аргументы разделены нулевыми байтами (`\0`).
- `tr '\0' ' '` заменяет нули на пробелы, чтобы можно было искать подстроку.

### 7.5. Таймауты
- `read -t 1` – ждёт ввод не более 1 секунды.
- Используется в воркере для неблокирующей проверки канала.

## 8. Типичные ошибки и их решения

| Ошибка | Причина | Решение |
|--------|---------|---------|
| Воркер умирает при запуске через `nohup` | Использование `exec 3<>` (одновременное чтение/запись) или фоновой записи `echo ... >&3 &`. | Использовать два отдельных канала, читать с таймаутом, не использовать `&` после записи. |
| Диспетчер не видит запущенные воркеры | В `cmdline` может быть `./worker_1.sh` или `bash worker_1.sh`, а поиск ищет полный путь. | Искать по имени скрипта (`basename`), перебирая аргументы. |
| Оставшееся время не уменьшается | Переменная `REMAINING` в фоновом процессе не синхронизирована. | Убрать фоновый процесс, читать канал в основном цикле с таймаутом. |
| Каналы не создаются | Скрипт падает до `mkfifo` из-за синтаксической ошибки. | Проверить код на отсутствие лишних `;`, незакрытых кавычек, неправильных отступов. |
| Диспетчер не отвечает на сигналы | Неверный синтаксис `trap` или забыли указать `SIGINT` и `SIGQUIT`. | Использовать `trap '...' SIGINT SIGQUIT`. |

## 9. Дополнительные вопросы для углублённого понимания

1. **Что произойдёт, если воркер завершится аварийно (например, `kill -9`), а канал останется?**  
   – Канал останется в ФС, но при следующем запуске воркера с тем же PID (что маловероятно) возникнет конфликт. Обычно удаление каналов при старте (`rm -f`) решает проблему.

2. **Можно ли использовать `select` или `poll` в bash для работы с FIFO?**  
   – Bash не поддерживает `select` напрямую, но можно использовать `read -t` или внешние утилиты (`timeout`).

3. **Почему в `find_pid_by_script` используется `for word in $cmd`?**  
   – `cmd` содержит несколько аргументов (например, `bash ./worker_1.sh`). Разбиение на слова позволяет искать `worker_1.sh` независимо от его позиции.

4. **Как улучшить надёжность диспетчера, чтобы он не запускал несколько экземпляров одновременно?**  
   – Добавить блокировку через `flock` на файл-флаг.

5. **Зачем в воркере используется `tr -d '\r'` после чтения команды?**  
   – На случай, если команда была сгенерирована в Windows-среде и содержит символ возврата каретки.
